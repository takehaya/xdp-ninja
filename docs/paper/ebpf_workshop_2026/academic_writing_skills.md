# Japanese Academic Paper Review

## ワークフロー

1. 対象節の特定
2. 観点別 diagnosis (問題のある原則を報告)
3. 最初から全面 rewrite (部分パッチは提案しない)
4. 引用付き自己 scan -> 自己修正
5. 結果はレビューメモに書き出す。原稿ソースは直接編集せず、採否は著者が判断する

## 原則

原文を以下にしたがって都度修正するのではなく、意味を保ったまま全面リライトする。サブセクションの構造も必要に応じて見直す。

### 文体
- JTF 日本語スタイルガイドの常体で統一。体言止め禁止。
- 口語語彙排除 (「だいたい」「ちゃんと」「結構」「〜とか」「〜という話」)
- 実施したこと、その結果確認したことについては過去形で。それ以外は現在形で。
  - 評価 / 実験セクションは結果報告として過去形で書く。測定・確認・比較・観測の記述は「〜した」「〜できた」「〜であった」に倒し、節末のまとめ文も「〜を確認した」「〜が示せた」と過去形で締める。完了した測定を述べる framing の現在形 (「報告する」「示す」「比較する」「生成する」) も過去形にする。純粋な navigation (「以降の節では…示す」)、general truth (「verifier は version ごとに異なる」)、metric の定義は現在形のままでよい。
- () での補足説明禁止。ただし、表・参考文献等の参照での利用は許容される。
- 修辞的な `:` `;` `-` `**` を使わない。code 記法・URL・型表記・table 区切りなど機械的位置は問題なし
  - colon リスト構文 (「以下を扱う: X, Y」) は markdown 箇条書きに変える

### Paragraph writing

- **各 paragraph は umbrella topic sentence で「何の topic を扱う paragraph か」を冒頭で明示する。**
- 1-2 文しかない段落が連続する場合や、巨大な大段落がある場合は topic の切りかたが不適切であるため、段落の構成を見直し、適切な粒度の段落分けになるようにする。
- 何かを例示する場合は「例えば、」を使うなど、transition word の利用を推奨

### Academic Writing

- 論文は英訳前提のため、以下がはっきり把握できる文を書くこと。

1. 主語 (誰が / 何が)
2. 目的語 (何を)
3. 動詞 (具体的な action)

- 著者の行為は「我々は」を主語に立て、system の動作は対象 system 名を主語に立てる
- 動詞は「subject が object を何にする」が明確に説明できるものを使う
  - 悪い例: 「P4 を data-plane language ではなく protocol の定義集として用い、型検査、offset 計算に**接続した**。」何をどこに接続 (?) したのか意味不明
  - 悪い例: 「カーネルバイパスの DPDKは、本論文が扱うカーネル内のフィルタ表現の問題とは**直交する。**」直交するという動詞で何を伝えているのか不明
  - その他比喩の意味で 落とす、流す、寄せる、刺さる 等の動詞を用いているケース
- 願望や主観の含まれる表現を使わない
  - 「〜したい」「〜してほしい」「〜が望まれる」「需要が高まっている」→ 「〜する必要がある」「〜が求められる」
- 「〜と言える」「〜と思われる」は事実として書ける範囲は事実として書く
- 「主要な」「代表的な」等の曖昧な表現は、実例をいくつか提示するか、何%が、等の定量的な数字を記載する
- 借用英語慣用句 (「source of truth」「first-class」「seamlessly」「robust」「battle-tested」「production-grade」) は語源にフィットするか確認。削っても情報が落ちないなら飾り
- Hedge (「〜しやすい」「概ね正しい」「比較的高速」「うまく動く」「ほぼ X」「適度に」「十分に」) を次に置き換え
  - 二値判定: 「〜しやすい」→「〜される」「〜できる」
  - 数値: 「概ね正しい」→「N 件中 M 件で正しい」
  - 比較対象明示: 「比較的高速」→「実装 A と同帯」

### Section 構成

- サブセクション (サブセクションがない場合はセクション) の末尾には、そこでつまり何が言えたか・何が示せたかのまとめ文を必ず置く
- section opening では subsection への navigation を述べてよい
- それ以外、同じ section 内で同じ内容を 2 度書かない

### Preemptive defense 禁止

reader が誤解する前に否定する形 (「X は貢献ではない」「重要なのは…ではない」「主役は X ではない」) を本文に散らさない。scope statement は Limitation で 1 度だけ書く

### Abstract について

- 3 段落 (問題 / 提案 / 結果と結論)、350-450 字目安。結果と結論は過去形で。
- self contained であること。本文中で初めて定義される label 等を出さない。
- 必要十分であること。明示する必要のない固有名詞、提示する意味のない数字を出さない。
- 等幅 syntax でのコード例を入れない。inline も禁止。

### Abstract の Conclusion と Conclusion 節について

英訳前提の典型構造:

> 我々は X を [示した / 提案した / 実現した]。これにより Y が可能になった。

要件:
- 過去形で締める。能力主張の現在形 (「実行できる」「対応する」) は提案段落で使うもので結論には使わない
- Background と対 (背景で立てた問題提起と Conclusion の解決宣言が同じ scope の noun phrase で対応するのが望ましい)

## 自己 scan (引用付き、原則単位)

書き直し案を出した後、各原則について rewrite から quote を抜き出して照合する。「通過」「pass」ラベル単独は禁止する。

## 出力フォーマット

レビューは別のメモファイル (例: `review_memo/` 配下) に書き出す。原稿 (`.tex` 等) は直接編集しない。メモはレビュワーが「どこが・なぜ・どう変わるか」を追える形にする。

提案ごとに、次の3部構成で順番に書く:

1. **項目** — `file:line` アンカー・違反した原則・重大度を見出しに。
2. **Diff** — `diff` 言語指定のコードフェンスで示す。`-` の行が現状、`+` の行が提案、変更なしの文脈行は行頭にスペースを付けて変更箇所を周囲に固定する。動く行とその前後 1-2 行の文脈のみ示す。
3. **理由** — なぜその変更が成立し、意味が保たれるかを 1-2 文で。

Diff なしで併記するもの:

- 通過した原則 (直さないと判断した点とその理由)
- 未レビューの候補 (位置のみ・機械的な一括置換はしない)

全体は次の骨格に従う:

1. 観点別 diagnosis (問題のある原則のみ)
2. リライト方針 (短く、何を残し何を削るか)
3. 項目ごとのブロック (項目 / Diff / 理由) を提案の数だけ
4. 引用付き自己 scan
5. scan 結果に基づいた書き直し案の書き直し

## 注意

- 原稿ソースを勝手に編集しない。`.tex` を黙って書き換えるとレビュワーが何が動いたか追えず、レビュワーに優しくない。指摘はメモに書き出し、適用は著者に委ねる。
- パターンの未レビュー箇所 (例: すべての `, so`) を一括置換しない。各箇所をワークフローに通し、残りは位置のみ候補として記録する。
- 元の文が何を意味しているか不明な場合は、推測せずに user に質問する
- 書き直しの前に方針確認が必要な場面 (節の前提が動く、貢献の主張が変わる) では user に聞くこと
