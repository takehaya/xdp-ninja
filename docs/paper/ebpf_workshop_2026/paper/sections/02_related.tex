\section{Related Work}
\label{sec:related}

Prior work falls into three categories: expressive packet-filter
languages, P4-to-eBPF compilation, and verifier-safe DSL compilation.

Expressive packet-filter languages have explored protocol-aware
filtering beyond fixed L2--L4 headers. NetPDL/NetPFL connect
protocol descriptions with tunnel-aware filter
expressions~\cite{risso2006netpdl,ciminiera2010netpfl}.
pFSA/xpFSA model encapsulation structure as state transitions and
formalize filter composition and field
access~\cite{leogrande2015pfsa,cerrato2018xpfsa}. These systems
improve filter expressiveness for nested encapsulation and
protocol-specific fields. However, they assume execution over parsed
packet representations and do not generate eBPF programs for
in-kernel execution.

P4-to-eBPF systems focus on generating executable programs for the
Linux data path. p4c-ebpf and p4c-xdp compile P4 programs into
eBPF for Linux targets such as XDP and
tc~\cite{p4c_ebpf,tu2018p4cxdp}. Simon et
al.~\cite{simon2024honey} further combine P4 pipelines with
dynamically loaded eBPF programs. These systems generate
verifier-accepted eBPF programs for fixed protocol pipelines.
They do not address packet-filter expressions whose target fields
may appear behind nested encapsulation or within variable-length
protocol structures.

Recent work has argued that DSL compilers targeting the Linux
kernel should satisfy a verifier-safe
property~\cite{solleza2025verifiersafe}: every well-formed program
should compile into bytecode that the verifier accepts. Solleza et
al.~\cite{solleza2025verifiersafe} characterize this verifier-safe goal,
but do not address packet
filtering over nested encapsulation and variable-length protocol
structures.
